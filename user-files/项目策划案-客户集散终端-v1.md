# 客户集散终端 —— 项目策划案

> **项目代号**：Milogin's Prospector · 客户集散终端（Lead Hub）
> **版本**：v1.0（草案，待经理评审）
> **负责人**：项目中枢（本人）
> **日期**：2026-07-11
> **一句话定位**：以现有 Prospector 程序为引擎，建立"批量触达 → 询盘识别 → 价值分派 → 跟进催收 → 数据沉淀"的客户线索集散中枢，由中枢单点统筹，把优质线索按成交能力分发给不同销售跟进，全过程可记录、可复盘、可数据化。

---

## 0. 背景与定位

- **业务**：货代行业出口，主攻拉美市场，开发信语言英（EN）/ 西（ES）/ 葡（PT）。
- **痛点**：潜在客户触达靠人工，量小、易进垃圾箱；有客户回复（询盘）时响应慢、易漏接；线索分派凭感觉，大客户可能给了经验不足的销售；派出去就失联，跟进无记录；客户偏好散落在各人脑子和微信里，随人走失。
- **本项目定位**：**不新增/改造程序**，而是用现有能力组织出一套标准化业务链路。中枢（本人）是唯一的数据录入口与调度中心；销售不接触任何系统，只被动接线索、口头/消息回传反馈；程序负责触达、识别、存储与"该催谁"的自动盯梢。

## 1. 相关工具简介

### 1.1 核心引擎：Prospector 程序本体（Electron 桌面 mini-CRM）

| 模块 | 能力 | 在本项目中的作用 |
| :--- | :--- | :--- |
| **发信引擎** | 多账号矩阵、熔断保护、工作时段模拟、拟人化随机延迟、自动发送 | 规模化、低封号风险地批量投递开发信 |
| **收信引擎** | 多账号 IMAP 收信、AI 回复检测（Agnes）、退信识别 | 自动抓回复、AI 初判询盘、剔除退信/自动回复 |
| **客户开发引擎** | Apollo 搜索、智能邮箱推断、筛选器 UI | 持续补充潜在客户池，扩大触达面 |
| **数据底座** | SQLite 四表 CRM 模型：`companies` / `contacts` / `interactions` / `opportunities` | 线索、互动、商机、客户画像的统一存储与留痕 |
| **模板系统** | 多语言（EN/ES/PT）、阶段化（cold → F1…F4）、模板轮换 | 不同市场、不同跟进阶段的差异化触达 |

### 1.2 辅助工具（程序之外，零开发）

- **销售能力档**：一张极简对照表（A 档 ≈ 可成 100 万级 / B 档 ≈ 10 万级 / C 档 ≈ 常规），由中枢维护，作为分派依据。
- **现有沟通渠道**：微信 / 电话，用于中枢向销售**手动派单**与**手动催问**（公司未使用飞书，故不引入外部协作系统）。

> **数据承载核对**：本项目所需字段现有数据库**基本全部已具备**，无需改程序 —— `contacts.assignee`（派给谁）、`contacts.followup_note`（跟进反馈）、`contacts.opp_stage`（商机阶段）、`contacts.tags`（标签/偏好）、`opportunities.amount`（客户价值/金额）、`companies.score`（背调评分）、`inbox.type`（AI 询盘分类）、`interactions`（全量互动史）。仅"采购周期/决策角色"无专用列，用 `tags` 或 `followup_note` 记录即可。

## 2. 链路设计（单点中枢闭环）

### 2.1 六步闭环

```
① 批量触达        程序：多账号发开发信，阶段 cold→F1→F2…（send_log 留痕）
       ↓
② 自动收信 + AI 识别   程序：IMAP 收信，inbox.type 标"询盘"，打 important
       ↓
③ 中枢复核 + 估值    人工复核真伪 → 定客户价值（opportunities.amount / companies.score）
       ↓
④ 价值 × 能力 派单    填 contacts.assignee → 微信/电话手动通知对应销售
       ↓
⑤ 程序盯梢 + 手动催问   系统自动列出"超 X 天无反馈、该催"的线索 → 中枢手动催
       ↓
⑥ 反馈 + 偏好录入     中枢录入 followup_note / opp_stage / tags / interactions
       ↓
    数据沉淀 → 回哺下一轮触达与复购（闭环）
```

### 2.2 角色分工

| 角色 | 职责 |
| :--- | :--- |
| **程序（引擎）** | 批量触达、自动收信、AI 询盘初判、数据存储、"该催谁"自动盯梢 |
| **中枢（本人）** | 触达统筹、询盘复核、客户估值、派单决策、催问执行、反馈与偏好录入（**唯一数据入口**） |
| **销售（下级）** | 接收线索、跟进客户、口头/消息回传进展（**不接触任何系统**） |

### 2.3 关键规则

- **客户价值分级**：结合 `opportunities.amount`（预估金额）+ `companies.score`（背调评分）+ 询盘意向强度，分为 **S（高）/ A（中）/ B（一般）** 三级。
- **分派匹配**：S 级客户 → A 档销售，A 级 → B 档，B 级 → C 档（大客户绝不派给弱销售）。**公司销售资源充足，按线索价值动态调配，不设人数上限。**
- **催问触发**：`opp_stage` 处于"跟进中/报价中"且距 `updated_at` 超过阈值（建议 **3–5 天**）→ 自动进入催问待办清单。
- **询盘认定**：AI（Agnes）先标 → 中枢复核确认，剔除退信/自动回复/无效回复后才进入分派池。

### 2.4 客户偏好数据化（字段落位）

| 偏好维度 | 落位字段 |
| :--- | :--- |
| 产品/品类兴趣 | `contacts.category` / `companies.cargo_types` / `tags` |
| 采购量/预算 | `opportunities.amount` |
| 采购周期/决策角色 | `tags[]` 或 `followup_note`（标签化记录） |
| 语言/时区 | `companies.country`（推导语言时区） |
| 互动史 | `interactions` 表（全量收发记录） |

## 3. 实际解决的问题

| 原痛点 | 本方案如何解决 |
| :--- | :--- |
| **触达规模小、易进垃圾箱** | 多账号矩阵 + 熔断 + 工作时段拟人节流，规模化且低封号风险 |
| **询盘漏接、响应慢** | AI 自动识别 + 中枢复核，收件箱不用人肉盯，询盘不漏 |
| **分派凭感觉、资源错配** | 客户价值 × 销售能力匹配，好钢用在刀刃上 |
| **派出即失联、跟进黑箱** | `assignee` 留痕 + 自动盯梢待办 + 反馈录入，过程可视可追 |
| **客户数据随人流失** | 偏好结构化沉淀进库，成为公司资产，可复用、可复购 |

## 4. 发展目标（分阶段）

| 阶段 | 时间 | 目标 |
| :--- | :--- | :--- |
| **短期** | 1–2 月 | 跑通六步闭环，建立标准 SOP，规范数据入库口径，形成首月基线数据 |
| **中期** | 3–6 月 | 触达量与询盘量稳定爬坡，分派与催收机制成熟，客户画像持续积累 |
| **长期** | 6–12 月 | 数据驱动：用沉淀的偏好数据反向优化模板/选品/复购节奏，形成"获客—成交"可复制飞轮 |

---

## 5. 预期成果

- **流程成果**：一套标准化、可追溯的"客户开发—分派—跟进"闭环 SOP，脱离个人经验依赖。
- **数据成果**：一个持续增值的客户数据资产库（画像 + 互动史 + 商机），随人员流动不流失。
- **管理成果**：中枢对全流程线索状态一屏可视，分派与催收有据可查，便于经理复盘与考核。
- **量化成果**：见第 6 部分指标体系。

## 6. 项目验收量化报告

> 说明：以下目标值为**参考基准**，正式验收前以**首月运行的真实基线**校准。数据来源均对应程序内真实字段/日志，可导出、可核验。

### 6.1 指标体系

| 层级 | 指标 | 口径 / 数据来源 | 参考目标 |
| :--- | :--- | :--- | :--- |
| **触达** | 日均有效发信量 | `send_log`（status=sent） | ≥ 待定（按账号数×每日上限） |
| | 有效送达率 | 1 − 退信率（`contacts.is_bounced`/发送数） | ≥ 95% |
| | 覆盖客户/公司数 | `contacts` / `companies` 累计 | 月增 ≥ 待定 |
| **响应** | 回复率 | 有回复联系人 / 送达联系人（`interactions` direction=in） | ≥ 行业基准（校准后定） |
| | 询盘识别准确率 | AI 标询盘中复核确认为真的比例 | ≥ 85% |
| | 询盘响应时效 | 收到询盘 → 派单 中位时长 | ≤ 24 小时 |
| **分派** | 价值-能力匹配率 | 高价值客户派给对应档销售的比例 | ≥ 90% |
| | 派单及时率 | 复核确认 → 派单 ≤ 阈值 的比例 | ≥ 90% |
| **跟进** | 催问覆盖率 | 该催线索中实际催到的比例 | 100% |
| | 反馈回收率 | 派出线索中拿到销售反馈的比例 | ≥ 90% |
| | 跟进状态更新及时率 | `opp_stage` 按 SOP 及时更新的比例 | ≥ 85% |
| **转化** | 询盘 → 报价率 | 进入"报价中"的询盘比例 | 校准后定 |
| | 报价 → 成交率 | 成交商机 / 报价商机（`opportunities`） | 校准后定 |
| | 成交额 / 客单价 | `opportunities.amount` 汇总 | 校准后定 |
| **数据** | 客户画像完整度 | 关键偏好字段填充率 | ≥ 80% |
| | 有效客户库增长 | 非退信、有互动的客户数增长 | 月增 ≥ 待定 |

### 6.2 验收标准（阶段性）

- **短期验收（第 2 月末）**：闭环全流程跑通 ≥ 1 个完整周期；数据入库口径达标（画像完整度 ≥ 60%）；催问覆盖率 100%。
- **中期验收（第 6 月末）**：核心指标全部达到参考目标；至少产出 ≥ N 笔经本链路分派并成交的订单（N 由经理设定）。
- **长期验收（第 12 月末）**：形成可复制 SOP 文档 + 客户数据资产库；用数据驱动优化模板/选品，复购或二次触达转化可量化。
