**核心设计原则：**
- Python 服务只做「抓取 + 提取」，不做业务判断
- Electron 端管「调度 + 评分 + 存储 + UI」
- 所有接口无状态（状态由 Electron 端管理）

## 功能一：批量背调

### 用户场景

在客户列表勾选 10 家公司 → 点「批量背调」→ 看到进度条逐家完成 → 10 份报告一次性写入。

### 数据流

```
Renderer                          Main                          Scrapling
   │                                │                               │
   │ IPC: backcheck:batch-start     │                               │
   │ [c1, c2, c3, ..., c10]        │                               │
   ├───────────────────────────────>│                               │
   │                                │ 并发 HTTP (max 5)             │
   │                                │ GET /search/web?q=c1 ────────>│
   │                                │ GET /search/web?q=c2 ────────>│
   │                                │ GET /search/web?q=c3 ────────>│
   │                                │ GET /search/web?q=c4 ────────>│
   │                                │ GET /search/web?q=c5 ────────>│
   │                                │         （排队）               │
   │                                │<──── 每完成一家返回 ──────────│
   │ IPC: backcheck:progress        │                               │
   │ {company, status, percent}     │                               │
   │<───────────────────────────────│                               │
   │                                │                               │
   │ IPC: backcheck:batch-done      │                               │
   │ [{company, rating, report}]    │                               │
   │<───────────────────────────────│                               │
```

### 新增 IPC 通道

```
backcheck:batch-start  →  传入公司数组，返回 batchId
backcheck:progress     ←  实时推送 { batchId, company, progress, status }
backcheck:batch-cancel →  取消剩余任务
```

### Python 端：无新增接口

复用现有 `/search/web` 和 `/scrape/website`。并发由 Electron 端用 `Promise.all` + 信号量控制。

### Electron 端改动

```js
// 并发控制：最多 5 家同时进行
async function batchResearch(companies, concurrency = 5) {
  const results = [];
  const queue = [...companies];
  
  async function worker() {
    while (queue.length) {
      const company = queue.shift();
      const result = await researchOne(company);  // 复用现有单家逻辑
      notifyProgress(company.company, result);
      results.push(result);
    }
  }
  
  await Promise.all(Array.from({length: concurrency}, () => worker()));
  return results;
}
```

### 风险控制

| 风险 | 措施 |
|------|------|
| Scrapling 服务过载 | Electron 端限制并发数（默认 5），可在设置里调 |
| DDG 限流 | 每家搜索间隔 2-3 秒随机延迟 |
| 大批量超时 | 单家超时 30s，总超时不限制 |
| 中途关闭应用 | 恢复时检查未完成的公司，提示用户 |

## 功能二：客户自动发现

### 用户场景

输入「墨西哥 + 汽车零部件 + 进口商」→ 系统自动搜索潜在客户 → 返回结构化列表 → 可勾选加入客户库。

### 数据流

```
Renderer                     Main                       Scrapling
   │                           │                            │
   │ IPC: discover:search      │                            │
   │ {country, industry, ...}  │                            │
   ├──────────────────────────>│                            │
   │                           │ GET /search/discover       │
   │                           │ ?country=Mexico            │
   │                           │ &industry=auto-parts       │
   │                           │ &role=importer             │
   │                           ├───────────────────────────>│
   │                           │                            │
   │                           │    Scrapling Spider        │
   │                           │    并发搜索多源：           │
   │                           │    · DDG/Bing 组合搜索     │
   │                           │    · 贸易名录网站抓取       │
   │                           │    · 展会参展商页面         │
   │                           │    · 行业黄页               │
   │                           │                            │
   │                           │<─── [{company, url, ...}] ─│
   │                           │                            │
   │ IPC: discover:results     │                            │
   │ {companies: [...]}        │                            │
   │<──────────────────────────│                            │
```

### 新增 Python 接口

```
GET /search/discover
  参数:
    country:    string  (国家，必填)
    industry:   string  (行业/品类)
    role:       string  (importer / distributor / manufacturer / any)
    limit:      int     (返回数量，默认 20)
  返回:
    {
      ok: true,
      companies: [
        {
          company: "AutoPartes Mexicanas SA de CV",
          website: "https://autopartes-mx.com",
          country: "Mexico",
          category: "汽车零部件",
          snippet: "Founded in 1998, one of the largest auto parts importers in Mexico...",
          source: "trade-directory",
          confidence: 0.85
        },
        ...
      ],
      total: 45,
      sources: ["DDG search", "TradeIndia", "Expo Manufactura exhibitor list"]
    }
```

### 搜索策略（Python 端）

按优先级并联搜索：

| 优先级 | 来源 | 搜索方式 |
|--------|------|---------|
| P0 | DDG/Bing 组合搜索词 | `"Mexico" "auto parts" importer company` |
| P0 | 贸易 B2B 名录 | TradeIndia、Made-in-China buyer list |
| P1 | 行业展会参展商 | Expo Manufactura、Automechanika 官网 |
| P1 | Google Maps 公司列表 | 暂不实现（需 Google API） |

### Electron 端改动

搜索结果先展示，用户勾选后 → 调用 `contacts:import` 写入客户库（已有接口）。

## 功能三：决策人深挖

### 用户场景

背调完成后 → 点「查找决策人」→ 自动抓取官网 About/Team 页面 → 提取人名+职位+联系方式。

### 数据流

```
Renderer                       Main                      Scrapling
   │                              │                          │
   │ IPC: contacts:deepSearch     │                          │
   │ {website, company}           │                          │
   ├─────────────────────────────>│                          │
   │                              │ GET /scrape/contacts     │
   │                              │ ?url=xxx                 │
   │                              ├─────────────────────────>│
   │                              │                          │
   │                              │    1. 找 About/Team 页   │
   │                              │    2. 提取人员列表       │
   │                              │    3. 匹配决策角色       │
   │                              │    4. 推断邮箱格式       │
   │                              │                          │
   │                              │<── [{name, title, ...}] ─│
   │                              │                          │
   │ IPC: contacts:deepResults    │                          │
   │ {people: [...], source}      │                          │
   │<─────────────────────────────│                          │
```

### 新增 Python 接口

```
GET /scrape/contacts
  参数:
    url:  string  (公司官网 URL)
  返回:
    {
      ok: true,
      people: [
        {
          name: "Carlos Mendoza",
          title: "Supply Chain Director",
          department: "logistics",      // logistics / procurement / management / other
          email: "carlos.mendoza@company.com",  // 推断，非确认
          confidence: 0.7,
          source_page: "/about/management"
        },
        ...
      ],
      email_pattern: "firstname.lastname@company.com"  // 发现的邮箱格式
    }
```

### 角色匹配逻辑（Python 端）

提取到人名+职位后，按关键词归类：

```
物流/采购决策人:
  supply chain, logistics, procurement, compras, import, export,
  customs, shipping, freight, transportation, cadena de suministro,
  logística, adquisiciones, importación, exportación

高管（二级价值）:
  CEO, director, general manager, country manager, VP, president,
  director general, gerente general

其他: 跳过
```

### 邮箱推断逻辑

```
1. 从官网 HTML 中搜 @company.com 格式的真实邮箱
2. 统计已知邮箱的命名模式 (first.last / f.last / firstlast)
3. 对找到的决策人按相同模式推断
4. 标注 confidence 等级
```

## 功能四：变化监控

### 用户场景

一个背调过的老客户，系统每周自动重抓官网 → 发现有新招聘 → 推送通知「这家公司可能在扩张」。

### 整体流程

```
┌─────────────────────────────────────────┐
│  Main 进程（app 启动时 + 每小时定时器）    │
│                                          │
│  1. 读取监控列表（data/monitor-config.json）│
│  2. 筛选到期公司（距上次检查 >= 7天）      │
│  3. 逐家调用 /scrape/diff                │
│  4. 对比新旧快照 → 提取变化               │
│  5. 有变化 → 通知 + 写入报告              │
│  6. 更新检查时间                          │
└─────────────────────────────────────────┘
```

### 监控配置

```json
// data/monitor-config.json
{
  "companies": {
    "Vibracoustic Mexico": {
      "website": "https://vibracoustic.com.mx",
      "lastCheck": "2026-06-16",
      "lastSnapshot": "vibracoustic-mexico-2026-06-16.json",
      "interval": "weekly",         // weekly / biweekly / monthly
      "alerts": ["hiring", "expansion", "news"],
      "enabled": true
    }
  }
}
```

### 新增 Python 接口

```
GET /scrape/diff
  参数:
    url:         string  (目标网址)
    prev_snapshot: string  (上次抓取的文本，base64)
  返回:
    {
      ok: true,
      current: "当前页面文本",
      changes: [
        {
          type: "hiring",              // hiring / expansion / new_product / news / text_change
          severity: "high",
          summary: "新增 Supply Chain Manager 招聘",
          snippet: "...We are hiring a Supply Chain Manager to lead our Mexico operations..."
        },
        ...
      ],
      has_changes: true
    }
```

### 变化检测策略

| 检测类型 | 方式 |
|---------|------|
| 招聘信息新增 | 关键词匹配：hiring / career / join / vacancy / 招聘 |
| 扩张信号 | 关键词：new plant / expansion / 扩建 / 新工厂 |
| 新闻动态 | 对比新旧文本的差异段落 |
| 联系方式变更 | 邮箱地址变化检测 |

### 通知机制

```js
// 通知到托盘 + 主界面
if (alert.type === 'hiring' && alert.severity === 'high') {
  new Notification({
    title: `🔔 ${companyName} 有新招聘`,
    body: alert.summary,
  }).show();
}
```

## 实现优先级

| 优先级 | 功能 | 工作量（估） | 理由 |
|--------|------|:--:|------|
| **P0** | 批量背调 | 小 | 复用现有接口，只加并发调度 |
| **P0** | 决策人深挖 | 中 | 官网抓取已就绪，加页面导航+提取 |
| **P1** | 客户自动发现 | 中 | 需要设计搜索策略 + 结果清洗 |
| **P2** | 变化监控 | 大 | 需要调度器 + 快照存储 + 对比引擎 |

---

## 不做的

- 不存用户隐私数据到 Python 服务（无状态设计）
- 不做实时监控（变化监控是定时轮询，不是 push）
- 不深挖邮箱验证（推断即可，验证是发送阶段的事）
- 不爬需要登录的网站（合规边界）
